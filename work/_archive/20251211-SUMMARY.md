### 4. 重要な発見と学習

#### ACM証明書のリージョン要件
**重要**: CloudFront用のACM証明書は**必ずus-east-1リージョン**で作成する必要がある
- 理由: CloudFrontはグローバルサービスで、証明書はus-east-1のみ参照可能
- ドキュメント確認により事前に把握し、正しく実装

#### Route53 Aliasレコード
- CloudFrontのHostedZone IDは固定値: `Z2FDTNDATAQYW2`
- Aliasレコードは追加料金なし（通常のDNSクエリ課金のみ）

#### Origin Access Control (OAC) vs OAI
- OACは新しい推奨方式（2022年導入）
- S3バケットポリシーでCloudFrontサービスプリンシパルを許可
- Condition句で特定のDistributionのみアクセス可能に制限

# 追加作業: EKS業務システム クラスタ構成比較分析

## 📅 作業情報
- **作業日**: 2025-12-11 (JST)

- **ステータス**: ✅ 完了

## 🎯 分析目的

3つの業務システムをAWS EKS上にデプロイする際の、以下2パターンの比較分析：
- **Pattern A**: 1つのEKSクラスタ + Namespace分離（ソフトマルチテナンシー）
- **Pattern B**: 3つの独立したEKSクラスタ（ハードマルチテナンシー）

## 📁 生成ファイル

### 主要成果物
- `EKS業務システム_クラスタ構成比較分析.md` - 包括的なアーキテクチャ比較分析ドキュメント

## 🔍 分析内容

### 1. コスト比較（東京リージョン ap-northeast-1）

| パターン | 年間コスト | 内訳 |
|---------|----------|------|
| **Pattern A** (1クラスタ) | **¥11,129** | コントロールプレーン: ¥8,760<br>ワーカーノード: ¥2,190 (t3.medium × 3)<br>NAT Gateway: ¥219 |
| **Pattern B** (3クラスタ) | **¥33,388** | コントロールプレーン: ¥26,280<br>ワーカーノード: ¥6,570 (t3.medium × 9)<br>NAT Gateway: ¥438 |

**コスト差**: Pattern B は Pattern A の **約3倍** (約¥22,259/年の追加コスト)

### 2. 運用負荷比較

| 観点 | Pattern A (1クラスタ) | Pattern B (3クラスタ) |
|-----|---------------------|---------------------|
| **運用対象** | クラスタ1個 | クラスタ3個 |
| **アップグレード作業** | 年1-2回 | 年3-6回（3倍） |
| **監視設定** | 1クラスタ分 | 3クラスタ分（3倍） |
| **バックアップ** | 1セット | 3セット（3倍） |
| **トラブルシューティング** | 1つの環境調査 | 3つの環境調査が必要 |

### 3. セキュリティ比較

#### Pattern A のセキュリティ実装
- **Namespace分離**: Kubernetes Namespaceによる論理分離
- **Network Policy**: デフォルト deny-all + 必要最小限の通信許可
- **RBAC**: Namespace単位のきめ細かな権限管理
- **Resource Quota**: Namespace単位のリソース制限
- **Pod Security Standards**: restricted レベルの強制

**重要な発見**: AWS公式ドキュメントによると
> "The cluster is the only construct that provides a strong security boundary"

ホストレベルの侵害が発生した場合、同じクラスタ内のすべてのSecrets/ConfigMaps/Volumesにアクセス可能。これがPattern Aの主要なセキュリティリスク。

#### Pattern B のセキュリティ実装
- **完全な物理的分離**: クラスタレベルでの完全分離
- **独立したコントロールプレーン**: 各システムが独自のK8s API Serverを持つ
- **ネットワーク完全分離**: 異なるクラスタ間は完全に分離
- **侵害範囲の限定**: 1つのクラスタが侵害されても他のシステムに影響なし

### 4. 推奨パターン

#### ✅ Pattern A を推奨するケース（一般的な業務システム）
- コスト重視（年間¥22,259の削減）
- 運用効率重視（管理対象1/3に削減）
- 中程度のセキュリティ要件
- 開発環境・ステージング環境

#### ✅ Pattern B を推奨するケース（高セキュリティ要件）
- 金融システム、医療システム
- コンプライアンス要件（規制対応）
- 顧客データの完全分離が必須
- SLA要件が異なるシステム

## 💡 学習ポイント

### 1. コスト・パフォーマンス分析
- EKSコントロールプレーンは固定費（$0.10/時）
- 小規模システムでは複数クラスタの固定費が大きな負担
- ワーカーノードの効率的な共有がコスト削減の鍵

### 2. セキュリティ境界の理解
- **クラスタレベル**: 最強のセキュリティ境界
- **Namespace + Network Policy**: 実用的な分離レベル
- セキュリティ要件に応じた適切な境界選択が重要

### 3. 運用スケールの影響
- クラスタ数の増加 = 運用負荷の線形増加
- 小規模チームでは運用負荷が重要な判断基準
- 自動化の導入で運用負荷は軽減可能

### 4. アーキテクチャ判断基準
6つの評価軸でパターンを比較：
1. コスト効率
2. 運用負荷
3. セキュリティレベル
4. スケーラビリティ
5. 障害影響範囲
6. 柔軟性
